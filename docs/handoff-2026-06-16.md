## 3. Outbox-relay（v0.9.0）— 本セッションの主成果・本番稼働中

### 3.1 アーキテクチャ（ADR 0020、追加固定費ゼロ）
```
ECS server (常駐)                     Lambda (event-driven)
┌─────────────────┐    SQS    ┌──────────────────────────┐
│ relay goroutine │──publish─▶│ consumer                 │
│ 10秒ポーリング   │  (direct) │ ① DynamoDB 条件付PutItem  │ 冪等
│ outbox_events   │           │ ② SES SendEmail          │
└─────────────────┘           └──────────────────────────┘
```
**ゼロ円を実現したキー判断**:
- relay は **ECS server 内の goroutine**（既存 egress 再利用 → NAT 不要）
- **SNS を省略**（relay → SQS 直接）
- Lambda は **VPC 外**（← 最大のコスト罠 NAT Gateway 月 $32 を回避）
- dedup は **DynamoDB + TTL**（SODA ブログの「揮発性 DB + TTL」パターンを実践、VPC 不要・無料枠）
- まずは **payment_succeeded のみ**対応

### 3.2 実装ファイル（Go）
| ファイル | 内容 |
|---|---|
| `payment/internal/domain/publisher.go` | `Publisher` port + `PublishMessage{EventID, Type, Email}` |
| `payment/internal/usecase/relay_outbox.go` | `RelayOutbox`（FindPending → 種別判定 → payment/email 解決 → Publish → MarkPublished、batch 50、publish 失敗は continue で pending 維持） |
| `payment/internal/infra/sqs_publisher.go` | `sqsPublisher`（json.Marshal → SendMessage） |
| `payment/payment.go` | `RunRelay`（10 秒 ticker、`relayEnabled = SQSQueueURL != ""`、未設定は no-op） |
| `cmd/server/main.go` | `sqsClient` 生成 + `go paymentModule.RunRelay(ctx)` |
| `valueobject/outbox_event_type.go` | `IsPaymentSucceeded()` |
| `cmd/outbox-consumer/main.go` | **Lambda consumer**（別 main パッケージ = 別バイナリ） |

### 3.3 冪等性（核心・SODA 決済で重要）
- **重複は 3 経路から来る**: relay 再送 / SQS at-least-once / Lambda リトライ
- **対策**: `cmd/outbox-consumer/main.go`
  ```go
  ConditionExpression: aws.String("attribute_not_exists(event_id)")
  ```
  DynamoDB の組み込み関数。**書き込みと存在チェックが原子的**（SQL の `INSERT ... IF NOT EXISTS` 相当）。同時実行でも必ず 1 つだけ成功。
- 失敗時 → `ConditionalCheckFailedException` → `return nil`（SES 呼ばずスキップ）
- **SES 送信失敗時は `DeleteItem`** → 印を消して再送可能に（「印は付いたが未送信」の穴を防止）

### 3.4 インフラ（Terraform、本番 apply 済み）
- `modules/sqs`: `musclead-outbox` + DLQ（maxReceiveCount=5）
- `modules/ecs`: `OUTBOX_QUEUE_URL` env + SQS SendMessage IAM ポリシー追加
- `modules/outbox_consumer`:
  - DynamoDB `musclead-outbox-dedup`（PAY_PER_REQUEST + TTL `expires_at`）
  - IAM role（assume lambda）+ basic/sqs/app ポリシー（app = PutItem/DeleteItem on dedup + ses:SendEmail）
  - Lambda（`provided.al2023` / arm64 / handler `bootstrap`）
  - `event_source_mapping`（batch 10、window 5s）= **②の SQS→Lambda トリガー（コードでなくインフラ設定）**
  - SES identity + **DKIM CNAME（`count = 3` で Route53）** ← `for_each` 不可（token は apply 後確定 = plan 時 unknown）

**本番リソース**:
- SQS: `https://sqs.ap-northeast-1.amazonaws.com/204340689570/musclead-outbox` + DLQ
- Lambda: `musclead-outbox-consumer`（Active）
- DynamoDB: `musclead-outbox-dedup`
- SES: `musclead.com`（Verified / DKIM SUCCESS）、送信元 `no-reply@musclead.com`

## 5. 技術的学び（SODA 実務直結）
- **Outbox パターン**（dual-write 問題）: 業務変更と同 TX で outbox 行を書く → atomic
- **at-least-once + 冪等性**: 条件付き書き込みで重複排除（決済の超重要パターン）
- **AWS Lambda + Go**: `provided.al2023`、バイナリ名 `bootstrap` 固定、`GOOS=linux GOARCH=arm64 CGO_ENABLED=0`、zip
- **AWS SDK v2 パッケージ分割**: `service/X`（Client + Input/Output）vs `service/X/types`（モデル型）。混同しやすい
- **Event Source Mapping**: SQS→Lambda は **AWS 設定（Terraform）、Go コードではない**
- **別バイナリ**: `cmd/server`（ECS 常駐）と `cmd/outbox-consumer`（Lambda）は同一 repo/go.mod だが別 main = 別アプリ、SQS のみで通信
- **DynamoDB**: 条件式 `attribute_not_exists` / `attribute_exists`、TTL 自動削除、PutItem は条件無しだと上書き
- **Terraform**: `count` vs `for_each`（apply 後確定値は for_each 不可）、`var.X`(外部入力) vs `aws_X.Y.arn`(リソース参照=自動)、`-target` apply、`path.module`
- **NAT Gateway コスト罠**: Lambda を VPC 外に出して回避
- **SES**: ドメイン検証 + DKIM（到達率）、サンドボックス（検証済み宛のみ）
- **React**: 厳格 lint 下での「レンダー中 state 調整」パターン
